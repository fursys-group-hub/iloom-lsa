### 3. 수정 패턴 요약

| 기존 패턴 (위험) | 새 패턴 (안전) |
|---|---|
| `new Date().toISOString().slice(0, 10)` | `getKSTToday()` 또는 `new Date().toLocaleDateString('sv-SE', { timeZone: 'Asia/Seoul' })` |
| `new Date(utcStr).toISOString().slice(0, 10)` | `toKSTDateString(utcStr)` 또는 `new Date(utcStr).toLocaleDateString('sv-SE', { timeZone: 'Asia/Seoul' })` |
| `new Date(now + 9*60*60*1000).toISOString().slice(0, 10)` | `new Date().toLocaleDateString('sv-SE', { timeZone: 'Asia/Seoul' })` |
| `.toLocaleDateString('ko-KR', { month: ... })` | `.toLocaleDateString('ko-KR', { timeZone: 'Asia/Seoul', month: ... })` |

## 🤔 결정한 것:
- **엑셀 시리얼 날짜 파싱** (`attendance-parser.ts`의 `excelSerialToDate`) — 이미 로컬 Date 객체 기반이라 수정 불필요
- **`new Date(date + 'T00:00:00')` 패턴** — 클라이언트에서 로컬 Date 생성이므로 한국 브라우저에서는 안전. 수정 불필요
- **API 타임스탬프** (`updated_at: new Date().toISOString()`) — DB 저장용 UTC 타임스탬프는 올바른 사용법. 수정 불필요
- **practice-report API** — 이미 `+09:00` 명시 사용중. 수정 불필요
- `toLocaleDateString('ko')` 호출에 모두 `timeZone: 'Asia/Seoul'` 추가 — 브라우저가 한국이 아닌 환경에서도 정확한 날짜 표시
